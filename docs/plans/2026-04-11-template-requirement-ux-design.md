# 模板诉求命名与编辑器紧凑检查器设计

**日期**: 2026-04-11  
**范围**: 模板结构命名、诉求编辑语义、模板编辑页 UX  
**状态**: 已确认，可进入实施规划

---

## 1. 目标

本次设计解决两个问题：

1. 模板“用户层语义”命名不稳定，`outline / blueprint / block / requirement` 混用，导致实现和界面表达不一致。
2. 模板编辑页当前是偏“大卡片表单”的组织方式，信息密度低，比例失衡，短字段占位过大，长模板编辑效率差。

本次不扩张功能边界，只做：

- 统一命名模型
- 统一用户可见术语
- 重构模板编辑页为更专业、更紧凑的检查器式编辑器

---

## 2. 核心语义模型

### 2.1 语义分层

命名基线固定如下：

- `outline`
  - 章节级容器
  - 表示诉求树、诉求确认对象、章节级结构
  - 必须保留，不能被 `requirement` 替代
- `requirement`
  - `outline` 节点中的单句诉求
  - 表示当前节点希望表达或分析的核心句子
- `slot`
  - `requirement` 句子中的变量位、可替换位、结构化输入位
  - 是技术实现命名，不直接暴露给最终用户

### 2.2 用户术语与技术术语

双层术语固定为：

- 用户可见
  - `诉求`
  - `诉求要素`
  - `诉求确认`
  - `实例级诉求树`
- 技术实现
  - `outline`
  - `requirement`
  - `slot`

原因：

- `outline` 是结构容器，保留最稳
- `requirement` 准确描述“每一句”
- `slot` 比 `block / item / requirement_item` 都更贴近真实语义
- 用户界面继续显示“诉求要素”，避免暴露技术词 `slot`

### 2.3 推荐字段映射

模板章节的用户层建议收敛为：

```json
{
  "outline": {
    "requirement": "分析 {@target_scene} 在 {@period} 内的核心趋势",
    "slots": []
  }
}
```

实例级诉求建议收敛为：

```json
{
  "requirement_instance": {
    "requirement_template": "分析 {@target_scene} 在 {@period} 内的核心趋势",
    "rendered_requirement": "分析 总部 在 最近7天 内的核心趋势",
    "segments": [],
    "slots": []
  }
}
```

### 2.4 占位符建议

现有 `{@block_id}` 不再推荐继续扩展。

建议统一到：

- `{@slot_id}`

原因：

- 变量位本质是槽位，不是 block
- `{@requirement_item_id}` 太长，读写成本高
- `{@item_id}` 过于泛化，不如 `slot` 清晰

## 3. 命名切换策略

### 3.1 哪些名称保留

以下名称保留，因为它们表达的是“容器/流程”而不是“句子”：

- `outline`
- `review_outline`
- `outline_override`
- `outline_content`
- `outline tree`

### 3.2 哪些名称迁移

以下名称迁移到新语义：

- `document` -> `requirement`
- `rendered_document` -> `rendered_requirement`
- `document_template` -> `requirement_template`
- `blocks` -> `slots`
- `block_id` -> `slot_id`
- `block_type` -> `slot_type`
- `outline_instance` -> `requirement_instance`

### 3.3 迁移原则

- 不把 `outline` 改没
- 不再使用 `block` 作为业务概念
- 用户界面全面切换为“诉求 / 诉求要素”
- 技术实现逐步切换为 `outline + requirement + slot`

---

## 4. 模板编辑页现状问题

当前编辑器的主要问题不是配色本身，而是信息架构和面积分配：

1. 左中右三块信息权重不清楚，很多内容像平铺表单，而不是结构化编辑器。
2. 诉求要素编辑采用“大块卡片”方式，随着要素增多，页面高度膨胀明显。
3. 短字段和长字段使用了相近的宽度策略，造成空间浪费。
4. 动作层级不清楚，新增、删除、上下移动、保存都容易形成视觉噪音。
5. 当前 tabs 和卡片的存在感偏强，削弱了编辑器的专业感。

## 5. UX 方向

### 5.1 总体方向

编辑页风格收敛为：

- 专业配置台
- 轻量检查器
- 信息密度适中但更紧凑
- 结构优先，不靠大面积卡片制造层次

不做：

- 全屏画布式编排器
- 重型拖拽布局系统
- 视觉炫技式重做

### 5.2 推荐方案

采用三栏“导航 + 主编辑区 + 检查器”的结构。

#### 左栏

定位：导航与切换  
建议宽度：`240-280px`

内容：

- 参数列表
- 章节树
- 当前选中状态

要求：

- 更像资源树，而不是按钮墙
- 每一行只展示摘要，不展开大字段
- 选中态清晰，但 hover/selected 都要克制

#### 中栏

定位：主编辑区  
占据主要空间

内容：

- 参数主字段编辑
- 诉求定义
- 执行链路主编辑
- 文稿和 SQL 等长字段

要求：

- 稳定，尽量减少跳动
- 长文本、模板文稿、SQL、prompt 继续 full-width
- 当前操作焦点应始终在中栏

#### 右栏

定位：检查器  
建议宽度：`300-360px`

内容：

- 诉求要素详情
- 同步状态
- 引用摘要
- 预览值
- 校验提示

要求：

- 高密度、轻量
- 不是第二个主表单区
- 只承载当前选中项的次级属性与上下文信息

## 6. 参数工作台重构

参数工作台保留当前功能，但改组织方式。

### 6.1 左侧

显示参数列表：

- 参数名称
- 类型
- 必填状态

### 6.2 中间主编辑区

显示核心字段：

- `label`
- `说明`
- `options/source`
- 默认交互语义

### 6.3 布局规则

短字段压缩在同一行：

- 参数标识
- 输入类型
- 追问模式
- 多值
- 必填

中字段用半宽：

- `source`
- `param_id`

长字段 full-width：

- 说明
- 复杂选项文本

---

## 7. 章节与诉求工作台重构

### 7.1 章节树

章节树继续保留在左栏，但改成更接近编辑器资源树：

- 标题
- 层级缩进
- 小型状态摘要
  - 是否有诉求
  - 是否有执行链路
  - 是否待同步

### 7.2 中栏默认视图

章节选中后，中栏默认显示：

1. `诉求定义`
2. `执行链路`
3. `结构预览`

### 7.3 诉求要素编辑

这是本次 UX 改造的重点。

现状问题：

- 每个要素都是一个大表单卡片
- 7 个要素就会拉出一整屏
- 很难先看结构，再聚焦编辑某一个要素

改造后：

- 中栏只展示“诉求要素列表”
- 每一行显示：
  - `ID`
  - `类型`
  - `提示`
  - `来源摘要`
- 点击某一行后：
  - 右栏打开该要素的检查器
  - 在右栏编辑详细属性

效果：

- 主区域先看全局结构
- 细节编辑集中在右侧
- 页面不会被重复卡片撑高

### 7.4 执行链路

执行链路改成 3 个可折叠区：

- `数据集`
- `展示方式`
- `引用绑定`

默认只展开当前正在编辑的部分，减少无效信息占位。

---

## 8. 视觉密度与交互规则

### 8.1 密度规则

- 卡片内边距整体下调一档
- 列表项高度下调
- 分组间距缩小
- tabs 改成更轻的 segmented control
- 短字段优先单行布局
- 长字段继续 full-width

### 8.2 交互规则

- 顶部主动作只保留 `保存模板`
- `新增 / 复制 / 上移 / 下移 / 删除` 降级为局部工具条或 icon button
- 危险动作与日常动作分开
- hover 效果更轻，selected 更明确
- 减少“胖按钮 + 胖标签 + 胖卡片”的重复堆叠

### 8.3 视觉风格关键词

- professional
- inspector-like
- compact
- structured
- restrained

---

## 9. 实施顺序建议

### Phase 1：命名模型落地

- 保留 `outline*`
- 引入 `requirement` 和 `slot`
- 统一用户文案为“诉求 / 诉求要素”
- 对现有类型与序列化层完成字段映射

### Phase 2：模板编辑页骨架重构

- 三栏比例重构
- 左栏资源树化
- 中栏主编辑区稳定化
- 右栏检查器落位

### Phase 3：诉求要素编辑改造

- 从大卡片表单改为列表 + 检查器
- 把要素详情迁移到右栏

### Phase 4：视觉密度收口

- 调整间距、tabs、工具条、按钮层级
- 收敛 hover / selected / 卡片存在感

---

## 10. 成功标准

完成后应满足：

1. 用户和开发者都能清楚区分：`outline` 是容器，`requirement` 是句子，`slot` 是变量位。
2. 模板编辑页明显更紧凑，短字段不再占据过大面积。
3. 诉求要素编辑不再依赖大块重复卡片。
4. 页面整体更像专业配置台，而不是堆叠表单。
5. 不改变现有核心工作流，只优化命名和编辑体验。
