# 通用对话

通用对话用于承载用户与系统之间的连续交流。报告生成是其中一种业务场景，但普通会话本身不依赖报告模板。

## 用户可以做什么

- 创建新的会话。
- 查看历史会话和消息记录。
- 查看由平台托管的历史会话和消息记录。
- 在系统处理较长任务时查看持续更新的进度。

## 对话体验

- 用户发送问题后，可以逐步看到系统正在处理的阶段。
- 当系统正在处理较长任务时，用户可以看到实时进展和局部结果。
- 对于可以拆分的任务，系统可以并行处理多个部分，但前台看到的进度和结果仍保持有序。
- 对于仍在运行中的任务，用户可以发起取消。
- 如果用户在任务运行中继续输入普通补充说明，前台会先显示为排队中，等当前轮结束后再作为新一轮对话发送。
- 用户通常只需要直接描述需求；系统会识别应该进入的业务场景。
- 当系统无法可靠判断用户意图时，会先请用户补充说明，而不是擅自进入某个业务流程。
- 系统需要澄清意图或补充信息时，会明确提出问题并等待用户回答。
- 用户回答已形成的结构化追问后，系统会将答复关联到对应追问，再以新一轮消息继续推进当前业务场景。
- 同一段连续交流会优先延续已经进入的业务场景，避免补充信息时意外切换流程。
- 已经完成的结果会保留在对应会话中，便于后续追溯。
- 历史记录按“一轮提问 + 多段回答”展示；同一轮中可以先显示文本提示，再显示进度、追问、局部结果、最终结果或错误。
- 会话记录按当前用户隔离，不会与其他用户混在一起。
- 用户身份由平台统一管理；报告系统只使用平台已经确认的当前用户身份。
- 用户输入和系统回答都会经过安全检查；存在风险时，系统会拒绝继续处理并给出提示。
- 系统在处理过程中发现无法继续、安全风险或条件不满足时，可以主动停止当前任务或拒绝回答，并说明原因。
- 系统返回失败时，应提供稳定的问题标识和用户可理解的提示。例如当前会话仍在处理时，应提示用户等待当前轮完成后再继续，而不是返回不明原因的系统异常。
- 系统会在任务结束时汇总资源用量和关键处理指标，用于平台运维、成本分析和质量观察；这些指标不会替代用户可见的对话结果。

## 场景扩展

不同业务能力可以复用通用对话。例如报告生成会引导用户补充条件、确认大纲，并逐步展示报告内容；智能问数会返回查询结果、解释和图表。

通用对话只负责承载追问和答复。具体询问什么、答复中包含哪些业务信息，由对应业务场景决定。例如报告生成会询问报告参数；未来其他场景可以使用完全不同的澄清内容。

系统内部也可以明确指定本轮希望进入的业务能力。未明确指定时，通用对话会根据用户输入和前序交流识别适合的场景；识别结果不够明确时，会返回通用澄清问题。

通用对话还负责把业务场景运行中的进展、外部工具处理状态、增量结果、最终结果和错误转换为前台可理解的实时消息。业务场景只描述自己的流程和结果，不直接处理前台流式协议。

报告场景的完整操作方式见 [报告生成](../report/generation.md)。

当前平台暂未开放删除历史会话和从历史消息创建分支的能力。
