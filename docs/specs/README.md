# ReportSystem 业务规格

本目录说明 ReportSystem 面向用户提供什么能力，以及这些能力如何使用。这里不描述代码结构、数据库表、接口报文或内部实现方式。

## 业务主线

用户可以围绕一个报告主题，选择或创建报告模板，通过对话补充生成条件，确认报告大纲，形成某次生成草稿，最终生成正式报告，并按需要导出 Word、PPT 或 PDF 文档。

```text
维护报告模板
  -> 在对话中提出报告需求
  -> 补充条件并确认大纲
  -> 生成和预览报告
  -> 调整局部内容
  -> 导出文档
```

## 统一原则

- 报告生成是通用对话中的一种业务场景；普通会话能力不依赖报告模板。
- 模板管理只维护可复用的报告模板，不在保存模板时直接生成报告。
- 报告管理只管理已经生成的正式报告，不承担模板识别和内容生成。
- 同一份正式报告用于页面预览、本地编辑和文档导出，避免不同入口看到不同结果。
- BI Engine 是页面预览和本地编辑的统一渲染内核。

## 结构定义与业务实例

这里的“定义”指系统设计开发时确定的结构定义，也就是 Schema。系统运行起来以后，业务用户通常不会修改这些定义；用户维护和使用的是符合这些定义的具体业务实例。

- 模板结构定义规定一份报告模板可以包含哪些条件、目录、页面、章节诉求和展示配置。
- 报告模板是符合模板结构定义的一份可复用配置资源，由业务人员维护。
- 生成草稿是某次生成时，系统基于报告模板、用户输入和外部数据形成的临时结果。用户确认前，它还不是正式报告。
- 报告结构定义规定一份报告可以表达哪些结构、组件、数据和生成证据。
- 正式报告是符合报告结构定义、并保存在报告中心的一份报告资源。它可以包含部分内容或完整内容，具体由报告状态区分。
- 生成完成后的正式报告可以用于页面预览、本地编辑和文档导出。

因此，Schema 是系统固定的定义；报告模板、生成草稿和正式报告都是运行中的业务对象。

## 术语映射

业务规格优先使用用户能理解的名称。研发阅读时，可以通过下表对应到内部术语。

| 业务名称 | 内部术语 | 说明 |
|---|---|---|
| 模板结构定义 | `ReportTemplate Schema` | 系统固定的模板结构约束 |
| 报告模板 | `ReportTemplate` | 符合模板结构定义的可复用配置资源 |
| 生成草稿 | `TemplateInstance` | 某次生成过程中基于报告模板形成的临时结果 |
| 报告结构定义 | `Report DSL Schema` | 系统固定的报告结构约束 |
| 正式报告 | `ReportInstance` | 保存到系统中的报告资源，承载报告正文、页面内容、来源和状态 |
| 导出文档 | `DocumentArtifact` | 从正式报告生成的 Word、PPT 或 PDF 文件 |

接口、Schema 和字段级技术手册见 [实现设计中的技术契约](../implementation/contracts/README.md)。
