# 测试实施计划

## 1. 目标

本次测试实施的目标不是做一次性冒烟，而是建立一套可持续复用的系统功能验证机制。计划同时覆盖：

- 文档化的正式用例基线
- 代码级自动化回归
- 真实服务的跨接口深度验证
- 测试证据沉淀与结果可追溯

## 2. 实施原则

- 以设计文档和当前公开接口为唯一范围基线
- 以自动化优先，人工只补不可脚本化或暂未脚本化的环节
- 以真实证据为准，不以“代码看起来正确”代替测试结论
- 报告必须同时包含覆盖结论、失败缺陷和剩余风险

## 3. 测试阶段

### 阶段 A：基线收敛

- 目标：
  - 固定正式测试范围
  - 识别设计文档与实现的差异
- 产出：
  - 本计划
  - 功能用例文档
  - 覆盖矩阵

### 阶段 B：自动化资产补齐

- 目标：
  - 补齐明显缺口
  - 为例行回归提供单命令入口
- 产出：
  - `/rest/dev` 路由回归测试
  - 报告中心与报告详情页回归测试
  - 深度 API 冒烟脚本
  - PowerShell 统一执行入口

### 阶段 C：代码级全量回归

- 目标：
  - 覆盖后端与前端的稳定自动化资产
- 执行项：
  - `python -m pytest src/backend/tests -q`
  - `npm test`
  - `npm run build`

### 阶段 D：真实环境深度测试

- 目标：
  - 验证真实服务、真实配置、真实外部接口下的系统行为
- 执行项：
  - 系统设置与重建索引
  - 模板创建、导入预解析、导出
  - 动态参数解析
  - 报告对话全链路
  - 报告聚合与文档下载
  - 设计文档与反馈接口

### 阶段 E：证据固化与报告输出

- 目标：
  - 生成测试报告
  - 记录通过项、失败项、风险与建议

## 5. 入口与退出标准

### 5.1 进入标准

- 后端服务可访问
- 前端页面可加载
- Completion/Embedding 配置可读取
- 数据库处于可读写状态

### 5.2 退出标准

- 代码级自动化全部执行完成
- 深度 API 冒烟执行完成
- 关键主链路结果明确：
  - 模板
  - 对话
  - 报告
  - 开发接口
- 测试报告完成并记录剩余风险

## 6. 自动化资产设计

### 6.1 代码级回归

- 后端：
  - 路由测试
  - 应用服务测试
  - 结构与边界测试
- 前端：
  - 页面测试
  - 组件测试
  - API 适配测试
  - 构建验证

### 6.2 深度 API 冒烟

脚本目标：

- 直接对真实后端发起请求
- 避免依赖浏览器手工操作
- 生成 JSON 结果，方便后续 CI 或例行任务消费

脚本覆盖：

- `templates`
- `chat`
- `reports`
- `parameter-options`
- `/rest/dev/design`
- `/rest/dev/system-settings`
- `/rest/dev/feedback`

### 6.3 统一执行入口

统一执行入口脚本负责串联：

- 后端全量测试
- 前端全量测试
- 前端构建
- 深度 API 冒烟

## 7. 风险与控制

### 7.1 已知风险

- 真实 LLM 与 Embedding 存在非确定性
- 当前设计文档之间仍存在旧口径残留
- 本地数据库含长期测试数据，可能影响列表型断言

### 7.2 控制措施

- 真实冒烟使用固定模板 ID 和唯一模板名
- 深度冒烟脚本对列表类断言采用“存在性”而非“数量唯一”
- 在报告中显式标记旧口径文档漂移

## 8. 证据保存

- 自动化命令输出保留在终端执行记录
- 深度 API 冒烟结果保存为 JSON
- 最终测试结论保存为 Markdown 报告

## 9. 执行顺序

1. 执行新增测试的小范围验证
2. 执行后端全量测试
3. 执行前端全量测试
4. 执行前端构建
5. 启动或确认真实服务
6. 执行深度 API 冒烟
7. 汇总结果，输出测试报告
