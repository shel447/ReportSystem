# 外部集成实现

本篇只记录非 Office Exporter 的外部依赖。Java 文档导出实现已独立进入 [文档导出实现](../report/document-export.md)。

平台外部接口的正式请求、响应和失败语义统一见 [外部依赖接口技术契约](../contracts/apis/external-dependencies.md)。本篇只记录适配器分层、缓存和调度等实现策略。

## 1. LLM 集成

LLM 能力只通过统一网关使用，应用层依赖能力 port：

- 模板匹配补充判断
- 参数提取
- 诉求实例化文案整理
- 章节内容生成
- 章节总结和报告汇总

这些能力属于报告生成场景，由 [报告生成实现](../report/generation.md) 编排。

## 2. 动态参数数据源

模板只声明 `source`。系统负责：

1. 将当前可见参数组装成标准请求体。
2. 向外部接口发送 `POST`。
3. 校验外部响应满足 `parameter-option-source-response.schema.json`。
4. 返回 `options + defaultValue`。

## 3. 平台外部能力

外部能力按职责拆分 adapter，不使用单个万能 HTTP gateway：

- Guardrail：问题、答案、SQL/Python 安全检查。
- AgentCore：会话、轮次、历史查询和 `chat/import` upsert。
- DataCatalog：逻辑实体、数据集和实体关系；Metadata Sync 作为辅助刷新接口，周期检查版本并触发缓存失效。
- Knowledge：知识查询、语义检索和多索引检索。
- OneQuery：执行 SQL/UQL。
- NodeAgent：启动与周期加载平台配置。
- Audit：非阻塞发送操作审计和安全审计。

底层 HTTP 统一使用外置 Runtime SDK 提供的 `runtime.client._session.GLOBAL_HTTP_SESSION`。ReportSystem 只传 `/rest/...` 相对路径，不读取或拼接平台服务 base URL。Runtime SDK 负责服务寻址、共享连接和底层认证；具体 adapter 负责身份透传、协议解析和 `chatbi.*` 错误转换。重试策略由具体 adapter 按接口幂等语义单独决定，不能在通用 POST client 中隐式开启。

`X-User-Id` 由上游网关完成认证后覆盖写入。ReportSystem 不保存用户资料，也不接受缺失或空白身份的正式业务请求。AgentCore、Guardrail、Parameter Options、Dynamic Custom、OneQuery、API Dataset、DataCatalog、Knowledge/RAG 和 Audit 外呼必须透传当前 `userId`。

## 4. 查询与执行

数据查询基础设施负责：

- 解释 `runtimeContext.bindings`。
- 运行 SQL 或 API 查询。
- 返回结构化结果和执行证据。

运行时禁止提前把多值参数拼成展示文本再送执行层；执行层消费三元组数组和最终 `resolvedQuery`。

生产 OneQuery 固定路径为 `/rest/dte/v1/onequery/uql/query`。报告 `sql/api` 数据集与智能问数复用 `data_analysis` 中的查询执行和元数据映射能力。

## 5. 业务配置、缓存和调度

- ChatBI 自己拥有的 AI、知识检索和数据分析配置由 [ConfigCenter](../configuration/README.md) 在启动时统一加载；NodeAgent appconf 只是其中一个配置来源。
- 平台服务地址、认证和连接参数属于 `runtime.client`，不进入 ConfigCenter。
- DataCatalog 默认缓存 600 秒；Knowledge 使用可替换 TTL cache port 的内存实现。DataCatalog 和 Knowledge/RAG 的缓存键包含 `userId`。
- `runtime.schedule` 在进程生命周期内周期检查元数据版本；ConfigCenter 本轮不周期刷新。当前调度线程为进程级 daemon，详见 [Runtime Schedule 接入](../scheduling/README.md)。
- 元数据更新后清理 DataCatalog、Knowledge 和模板语义索引缓存。
- 审计使用内存队列异步尽力发送；失败记录日志，不阻塞主业务。
