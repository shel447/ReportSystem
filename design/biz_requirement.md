## 一、业务流程设计



### 1、准备阶段

1.1、注册报告模板

### 2、运行阶段

2.1、用户访问对话（chat）接口，输入问题

2.2、系统接收到用户问题，根据用户问题匹配报告模板

2.3、根据报告模板的配置内容，提示用户补充必要与可选信息

2.4、用户确认后，系统根据报告模板与用户补充信息，生成报告实例，并在对话接口返回给用户预览、修改

2.5、用户修改后，可以对局部内容重新生成，更新报告实例

2.6、当用户最终确认内容无误后，生成报告文档

2.7、用户可以通过报告文档管理接口，对报告文档进行增删查改或者下载

## 二、关键概念

| 概念     | 定义                                                         | 补充说明 |
| -------- | ------------------------------------------------------------ | -------- |
| 报告模板 | 用户预先定义的报告模板。包括报告类型、报告场景、报告内容参数、报告大纲 |          |
| 报告实例 | 在报告模板的基础上，填充报告内容参数后，使用大语言模型技术栈生成报告实例。<br>报告实例是一种中间态，既能看到最终呈现样式，又保留着对报告模板具体内容以及参数的引用信息 |          |
| 报告文档 | 报告实例的物理载体，文档类型可以是word、pdf                  |          |

## 三、关键模块原始诉求

### 3.1、定时任务

**2026-03-02 团队讨论后新增的规格**

- 定时任务区分用户，用户看不到其他用户创建的定时任务，只能对自己管理自己创建的定时任务；

- 每个用户最多只能创建5个定时任务，系统全局最多100个定时任务；

- 定时任务需要支持一次性执行和周期性执行两种模式；

- 定时任务需要最终生成报告文档，不用等到手动触发

### 3.2、近期需求演进记录（按提出日期整理）

> 说明：以下内容用于保留近期迭代中陆续提出的新增需求与调整诉求，不覆盖上面的原始需求描述。

#### 2026-03-17

- 报告模板编辑从 JSON 直编逐步转为 UI 工作台方式；
- 对话生成报告流程调整为：模板匹配 -> 结构化补参 -> 参数确认 -> 大纲确认 -> 生成实例 -> 下载 Markdown；
- 参数收集需要严格按参数类型决定控件形态，日期类参数使用日期控件；
- 报告实例和系统设置等页面的公共壳层能力必须在所有页面保持一致，不允许新增页面后出现入口缺失；
- 报告实例支持生成和下载 Markdown 文档，暂不做 PDF；

#### 2026-03-18

- 前端整体视觉从“暗、重、闷”改为“明亮、专业、清爽”的风格；
- 前端架构从单文件静态页重构为 React + Vite + TypeScript；
- 模板管理改成“列表 -> 详情”的工作台模式，不再让列表和详情硬挤在同一屏；
- 对话助手主工作区以消息流为主，减少无效说明和重复标题；

#### 2026-03-19

- 对话输入框固定在页面底部并始终可见；
- 消息流尽量撑满工作区，减少右侧大块空白；
- 系统设置的操作反馈统一放到操作按钮组下方，短暂显示后自动消失；
- 模板工作台中参数区、章节区需要更清晰的分栏和分隔；
- 系统设置入口与“提意见”入口调整到统一壳层中的固定位置；

#### 2026-03-20

- 对话历史记录加入聊天页，位于全局侧边栏与聊天主区之间，支持折叠/展开；
- 会话标题由首条真实用户消息生成，进入 `/chat` 时不自动恢复最近会话，也不预创建空会话；
- 消息支持从历史消息 fork 新分支；
- 报告实例支持基于来源消息 fork，以及基于确认大纲“更新”出新会话；
- “模板实例”不再作为用户可见模块独立存在，而是内化为报告实例的内部生成基线；

#### 2026-03-30

- 对齐最新版 `ReportTemplate`，模板章节采用“双层模型”：
  - 面向用户的 `outline blueprint`
  - 面向系统的 `content / datasets / presentation`
- 对话助手的大纲确认改造成所见即所得内容树；
- 大纲确认支持 inline block 编辑，并区分：
  - 只改参数片段：不破坏蓝图
  - 改非参数文本：节点退化成普通句子
- 蓝图区块类型需要类型化配置与类型化控件，包括：
  - `time_range`
  - `indicator`
  - `scope`
  - `enum_select`
  - `param_ref`
  - `number`
  - `threshold`
  - `boolean`
  - `operator`

#### 2026-03-31

- 定时任务页恢复“新建任务”入口，且创建路径固定为“从已有报告实例创建”；
- 定时任务新增双时间模型：
  - `created_at` 表示真实执行时间
  - `report_time` 表示业务报告时间
- 定时任务支持把计划执行时间：
  - 写回实例输入参数
  - 作为实例 `report_time`
- 模板参数新增 `interaction_mode = form | chat`；
- 对话助手支持按参数顺序混排两类追问方式：
  - `form`：结构化面板追问
  - `chat`：自然语言追问
- 当报告流程正在等待 `chat` 模式参数时，普通自然语言优先作为参数答案，而不是被误判为智能问数或智能故障；
- 同一统一对话入口下支持三类能力：
  - 制作报告
  - 智能问数
  - 智能故障
- 当用户显式提出切换任务时，需要先通过聊天内确认卡片，避免误打断已有报告任务；
- 需要补齐 DFX 接口治理设计，统一异常响应、错误码、限流、分页、容量上限与长期保留策略；
- 需要系统化梳理模板、对话、实例、文档、定时任务、系统设置各模块的典型错误场景；
- 当前定时任务里“任务执行时间 / 报告时间 / 报告数据时间范围”的关系表达不清，需要作为后续专题单独设计；

#### 2026-04-10

- 本轮设计的核心动机，是把近几周快速演进后逐渐混杂的模型重新收敛，重点解决 4 类问题：
  - 报告模板、报告实例、内部生成基线的结构在表层字段上过于分散，不利于后续 schema 演进；
  - 会话、消息、用户归属、报告来源之间的边界不清，导致数据隔离、来源追溯和恢复逻辑别扭；
  - “蓝图”这一术语同时承担了用户可见表达、结构化占位和生成前中间态等多重含义，业务语义不稳定；
  - 对话接口虽然保留统一入口，但各阶段的请求/响应契约、结构化动作边界和流式策略还不够清楚。

- 因此，本轮总体思路不是继续做局部补丁，而是先在设计层统一口径，形成后续开发团队可以直接据此做重构、迁移和接口收敛的基线文档。

- 报告模板模型收敛为“诉求层 + 执行层”双层结构：
  - 面向用户的一层，统一用 `outline.document + outline.blocks[]` 表达章节诉求；
  - 面向系统的一层，继续使用 `content / datasets / presentation` 表达查询、推理与渲染链路；
  - 两层在同一章节节点共存，不再把“用户诉求”和“系统执行”拆成两个平行对象。

- 模板中的块配置不再以“蓝图区块”理解，而统一提升为“诉求要素”：
  - 参数仍然表示“系统要向用户收集什么”；
  - 诉求要素表示“这一段诉求由哪些对象、条件、范围、指标、维度、阈值等组成”；
  - 参数可以给诉求要素赋值，但二者不是同一个概念。

- 原先围绕 SQL 的值映射设计被重命名并中性化：
  - 不再使用 `sql_mapping`；
  - 改为 `value_mapping`；
  - 统一区分 `display / value / query` 三类值通道；
  - 这样可以覆盖 SQL 以外的执行场景，避免模板结构被某一种底层数据源技术绑死。

- 动态参数的数据源协议被正式定义：
  - 平台内统一增加 `POST /rest/chatbi/v1/parameter-options/resolve`；
  - 平台对外部数据源统一采用 `POST` 请求；
  - 统一规定请求体、响应体和规格上限：
    - 默认 `limit = 10`
    - 最大 `limit = 50`
    - 请求体上限 `32 KB`
    - 外部超时 `3s`
    - v1 不自动重试
  - 候选项格式统一收敛到 `[{label, value}]`，并保留对历史 `string[] options` 的兼容策略。

- 数据库模型做了系统性收敛，设计意图是“表顶层只保留过滤、排序、归属、追溯所需元字段，详细结构统一归入 `content`”：
  - `report_templates` 不再把参数、章节、匹配词等详细结构散落在表顶层；
  - `report_instances` 不再把输入参数、诉求树、生成内容拆成大量顶层列；
  - `template_instances` 作为纯内部概念，更明确地采用整体快照存储；
  - 三者都以 `schema_version + content` 的形式承载详细结构；
  - 这样做的策略，是把结构演进从“频繁改表”转为“升级 content schema”。

- 针对数据库范式，也单独做了设计口径补充：
  - 以 `1NF~3NF` 为主规范；
  - `4NF` 作为补充原则；
  - 明确 `schema_version + content` 属于“受控反范式化”；
  - 只允许在模板定义、实例内容、内部快照这类“整体对象”上使用，不允许变成任意塞 JSON 的兜底手段。

- 会话、消息、用户隔离和报告归属模型被重新梳理：
  - 引入 `tbl_users`，并约定 `X-User-Id` 为统一身份来源；
  - `chat_sessions` 收敛为会话容器；
  - 新增 `chat_messages` 表达真实消息流水；
  - 报告实例直接记录：
    - `user_id`
    - `source_session_id`
    - `source_message_id`
  - 会话不再反向挂报告实例，报告归属也不再依赖会话侧单字段表达。

- 这轮设计特别强调“报告实例是会话产物之一，但会话不是报告实例的宿主状态对象”：
  - 一个会话可以持续产生多个产物；
  - 报告实例只需记录它来自哪个会话、哪条消息；
  - 不应再让会话表反向维护单个 `instance_id`。

- `TemplateInstance` 与 `ReportInstance` 的关联方向也做了收敛：
  - 外键放在 `template_instances.report_instance_id`；
  - 不在 `report_instances` 上暴露 `template_instance_id`；
  - 动机是保持用户主对象干净，把内部生成基线作为依附对象处理；
  - 同时通过唯一约束表达“一份报告实例最多对应一份内部生成基线”。

- 表定义文档结构全面重做，作为后续开发和数据库迁移的主依据：
  - 先给 ER 图；
  - 再给表清单；
  - 每张表提供字段表格 + `CREATE TABLE IF NOT EXISTS` + 索引定义；
  - 统一约束主键字段名为 `id`；
  - 统一约束规范表名前缀为 `tbl_`；
  - 并明确当前代码实现表名与目标规范表名存在差异，正式落地前需要单独迁移方案。

- 文档术语体系从“蓝图”全面切换到“诉求”：
  - “诉求”表示用户希望系统获取并表达的信息意图；
  - “诉求要素”表示构成诉求的结构化成分；
  - “诉求实例”表示参数、局部变量和上下文绑定后的具体表达；
  - “生成报告”被明确降级为满足诉求的一种手段，而不是顶层概念本身。

- 术语切换策略不是简单全量替换字符串，而是做分层处理：
  - 总体设计文档重点讲业务概念边界；
  - 模块设计文档重点讲这些业务概念如何映射到当前结构字段、动作名和阶段名；
  - 因此 `outline / outline_instance / review_outline / outline_override` 仍可作为兼容字段名和动作名保留，但业务语义统一按“诉求”解释。

- API 文档与运行时文档也同步做了这一轮口径收敛：
  - 用户侧表述改为“诉求确认”“实例级诉求树”“确认诉求/生成基线”；
  - 同时补齐统一身份 Header、实例来源字段、动态参数解析接口等外围契约；
  - 但需要强调：对话接口的阶段化请求体/响应体、哪些环节采用流式输出、诉求修改和章节重生成的详细契约，目前仍是后续要继续深挖的专题。

- 对开发团队的直接影响和落地策略如下：
  - 后端改造时，应先按文档完成数据模型与接口语义收敛，再进入实现；
  - 尤其要注意用户隔离、来源锚点、消息流水拆表、`schema_version + content` 这几项是系统性改动，不适合零碎穿插修改；
  - 对已有数据，需要先做迁移方案、回填方案和兼容策略，再切换读写逻辑；
  - 对接口实现，需要把“业务语义名”和“兼容字段名”区分清楚，避免因为内部仍叫 `outline` 就在用户侧继续输出“蓝图/大纲”的旧概念；
  - 对验证团队，需要重点补齐以下验证面：
    - 不同用户之间的数据隔离
    - 报告实例来源追溯
    - 基于生成基线恢复 update-chat
    - 动态参数接口的限流、超时和异常语义
    - 模板/实例 `content` 结构与 `schema_version` 的一致性

- 本轮最终产出不是单个点状需求，而是一整套新的设计基线，供开发团队后续据此完成：
  - 数据库迁移设计
  - 接口契约细化
  - 会话与实例逻辑重构
  - 调试与回归验证
