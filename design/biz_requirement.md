## 一、业务流程设计



### 1、准备阶段

1.1、注册报告模板

### 2、运行阶段

2.1、用户访问对话（chat）接口，输入问题

2.2、系统接收到用户问题，根据用户问题匹配报告模板

2.3、根据报告模板的配置内容，提示用户补充必要与可选信息

2.4、用户确认后，系统根据报告模板与用户补充信息，生成报告实例，并在对话接口返回给用户预览、修改

2.5、用户修改后，可以对局部内容重新生成，更新报告实例

2.6、当用户最终确认内容无误后，生成报告文档

2.7、用户可以通过报告文档管理接口，对报告文档进行增删查改或者下载

## 二、关键概念

| 概念     | 定义                                                         | 补充说明 |
| -------- | ------------------------------------------------------------ | -------- |
| 报告模板 | 用户预先定义的报告模板。包括报告类型、报告场景、报告内容参数、报告大纲 |          |
| 报告实例 | 在报告模板的基础上，填充报告内容参数后，使用大语言模型技术栈生成报告实例。<br>报告实例是一种中间态，既能看到最终呈现样式，又保留着对报告模板具体内容以及参数的引用信息 |          |
| 报告文档 | 报告实例的物理载体，文档类型可以是word、pdf                  |          |

## 三、关键模块原始诉求

### 3.1、定时任务

**2026-03-02 团队讨论后新增的规格**

- 定时任务区分用户，用户看不到其他用户创建的定时任务，只能对自己管理自己创建的定时任务；

- 每个用户最多只能创建5个定时任务，系统全局最多100个定时任务；

- 定时任务需要支持一次性执行和周期性执行两种模式；

- 定时任务需要最终生成报告文档，不用等到手动触发

### 3.2、近期需求演进记录（按提出日期整理）

> 说明：以下内容用于保留近期迭代中陆续提出的新增需求与调整诉求，不覆盖上面的原始需求描述。

#### 2026-03-17

- 报告模板编辑从 JSON 直编逐步转为 UI 工作台方式；
- 对话生成报告流程调整为：模板匹配 -> 结构化补参 -> 参数确认 -> 大纲确认 -> 生成实例 -> 下载 Markdown；
- 参数收集需要严格按参数类型决定控件形态，日期类参数使用日期控件；
- 报告实例和系统设置等页面的公共壳层能力必须在所有页面保持一致，不允许新增页面后出现入口缺失；
- 报告实例支持生成和下载 Markdown 文档，暂不做 PDF；

#### 2026-03-18

- 前端整体视觉从“暗、重、闷”改为“明亮、专业、清爽”的风格；
- 前端架构从单文件静态页重构为 React + Vite + TypeScript；
- 模板管理改成“列表 -> 详情”的工作台模式，不再让列表和详情硬挤在同一屏；
- 对话助手主工作区以消息流为主，减少无效说明和重复标题；

#### 2026-03-19

- 对话输入框固定在页面底部并始终可见；
- 消息流尽量撑满工作区，减少右侧大块空白；
- 系统设置的操作反馈统一放到操作按钮组下方，短暂显示后自动消失；
- 模板工作台中参数区、章节区需要更清晰的分栏和分隔；
- 系统设置入口与“提意见”入口调整到统一壳层中的固定位置；

#### 2026-03-20

- 对话历史记录加入聊天页，位于全局侧边栏与聊天主区之间，支持折叠/展开；
- 会话标题由首条真实用户消息生成，进入 `/chat` 时不自动恢复最近会话，也不预创建空会话；
- 消息支持从历史消息 fork 新分支；
- 报告实例支持基于来源消息 fork，以及基于确认大纲“更新”出新会话；
- “模板实例”不再作为用户可见模块独立存在，而是内化为报告实例的内部生成基线；

#### 2026-03-30

- 对齐最新版 `ReportTemplate`，模板章节采用“双层模型”：
  - 面向用户的 `outline blueprint`
  - 面向系统的 `content / datasets / presentation`
- 对话助手的大纲确认改造成所见即所得内容树；
- 大纲确认支持 inline block 编辑，并区分：
  - 只改参数片段：不破坏蓝图
  - 改非参数文本：节点退化成普通句子
- 蓝图区块类型需要类型化配置与类型化控件，包括：
  - `time_range`
  - `indicator`
  - `scope`
  - `enum_select`
  - `param_ref`
  - `number`
  - `threshold`
  - `boolean`
  - `operator`

#### 2026-03-31

- 定时任务页恢复“新建任务”入口，且创建路径固定为“从已有报告实例创建”；
- 定时任务新增双时间模型：
  - `created_at` 表示真实执行时间
  - `report_time` 表示业务报告时间
- 定时任务支持把计划执行时间：
  - 写回实例输入参数
  - 作为实例 `report_time`
- 模板参数新增 `interaction_mode = form | chat`；
- 对话助手支持按参数顺序混排两类追问方式：
  - `form`：结构化面板追问
  - `chat`：自然语言追问
- 当报告流程正在等待 `chat` 模式参数时，普通自然语言优先作为参数答案，而不是被误判为智能问数或智能故障；
- 同一统一对话入口下支持三类能力：
  - 制作报告
  - 智能问数
  - 智能故障
- 当用户显式提出切换任务时，需要先通过聊天内确认卡片，避免误打断已有报告任务；
- 需要补齐 DFX 接口治理设计，统一异常响应、错误码、限流、分页、容量上限与长期保留策略；
- 需要系统化梳理模板、对话、实例、文档、定时任务、系统设置各模块的典型错误场景；
- 当前定时任务里“任务执行时间 / 报告时间 / 报告数据时间范围”的关系表达不清，需要作为后续专题单独设计；
