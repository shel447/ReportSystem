# 05. 数据模型与持久化

## 1. 持久化原则

1. 持久化命名统一采用 `conversation` / `chat`。
2. `ReportTemplate`、`TemplateInstance`、`Report DSL` 都有正式业务模型；数据库里的 JSON 列只是持久化承载，不再重新定义第二套结构。
3. 需要过滤、排序、关联的元字段单独建列；完整对象主体进入 `content`。
4. `content` 中保存的是完整正式对象本体，而不是额外再包一层 `content`。

### 2.6 `tbl_report_instances`

- 元字段：
  - `id`
  - `template_id`
  - `template_instance_id`
  - `user_id`
  - `source_conversation_id`
  - `source_chat_id`
  - `status`
  - `schema_version`
  - `created_at`
  - `updated_at`
- 主体：
  - `content JSON`

说明：

- `content` 的主体是完整 `Report DSL`
- `ReportInstance` 是资源概念；正式报告内容则由 `Report DSL` 承担
- `template_instance_id` 指向生成该报告时使用的模板实例快照来源
- 外键方向固定为 `report_instance -> template_instance`，而不是反过来
- 同一个 `template_instance_id` 可以派生多份 `report_instance`，因此这里不能加唯一约束

## 3. 归属与隔离规则

- `conversations`、`chats`、`report_instances`、`export_jobs` 必须按 `user_id` 强隔离
- `report_documents` 通过 `report_instance_id -> report_instances.user_id` 间接隔离
- 后续新增任何报告素材类表都必须带 `user_id NOT NULL`

## 4. 读写规则

1. 会话查询必须带 `user_id`
2. 聊天消息查询必须带 `user_id`
3. 报告查询必须带 `user_id`
4. 文档下载必须通过 `report_id + document_id` 做归属校验

## 5. 迁移要点

- 旧模板若把 `parameters/catalogs` 放在 `content` 子属性中，需要提升到正式对象根部
- 旧 `session/message` 命名统一迁移到 `conversation/chat`
- 旧 `outline_content/generated_content` 等运行态结构逐步回收到 `TemplateInstance` 与 `Report DSL` 的正式模型中
