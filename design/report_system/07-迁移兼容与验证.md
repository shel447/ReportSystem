# 07. 迁移兼容与验证

## 1. 迁移目标

本轮迁移的目标不是做增量补丁，而是把以下矛盾一次性收口：

- 模板与模板实例到底是 `sections/subsections` 还是 `catalogs -> (subCatalogs)* -> sections`
- 报告 DSL 到底是中间对象还是正式领域模型
- 模板与模板实例的正式结构是否额外包 `content`
- 统一对话接口和报告资源接口返回的报告载荷是否等价
- 会话与消息命名是否统一到 `conversation/chat`
- 报告实例与模板实例的关联方向是否唯一且清晰

## 2. 迁移顺序

1. 以本设计包的 Schema 为准，冻结正式模型
2. 回收旧模板 Schema，迁移到 `design/report_system/schemas/*`
3. 调整模板导入、模板校验、模板导出逻辑，统一到新 `ReportTemplate`
4. 调整模板实例读写，统一到新 `TemplateInstance`
5. 调整报告生成链路，统一 `TemplateInstance -> Report DSL`
6. 调整文档生成链路，只消费正式 `Report DSL`
7. 最后清理旧字段、旧命名和旧兼容层

## 3. 兼容规则

### 3.1 模板兼容

旧模板若存在：

- 顶层 `sections`
- `subsections`
- `content.parameters`
- `content.catalogs`

迁移时要统一规范化为：

- `parameters`
- `catalogs -> (subCatalogs)* -> sections`

### 3.2 模板实例兼容

旧模板实例若存在：

- `resolved_view`
- `outline_runtime`
- `generated_content`

迁移时要收敛为：

- `parameters`
- `catalogs`
- `deltaViews`
- `templateSkeletonStatus`

### 3.3 报告实例兼容

旧报告实例若仍保存：

- `outline_content`
- `input_params`
- `generated_content`

迁移时要收敛为：

- `Report DSL`
- `documents`
- 来源锚点字段

## 4. 验证要求

### 4.1 Schema 校验

- 模板导入前必须校验 `report-template.schema.json`
- 模板实例冻结前必须校验 `template-instance.schema.json`
- 报告实例冻结前必须校验 `report-dsl.schema.json`

### 4.2 接口验证

- `/chat` 确认参数后必须流式返回生成中的报告
- `GET /reports/{reportId}` 必须一次性返回完成态报告
- 二者的 `answer` 载荷必须结构等价
- `GET /reports/{reportId}` 的顶层 `reportId/status` 必须与 `answer.reportId/status` 一致
- `POST /reports/{reportId}/document-generations` 必须返回 `jobs + documents`

### 4.3 文档链路验证

- `word`、`ppt` 都从 `Report DSL` 生成
- `pdf` 由 `word/ppt` 派生生成
- 任一产物失败要可追踪到 `export_job`
