# 报告运行时实现

## 1. 模块定位

`report_runtime` 负责两个正式能力：

- `TemplateInstance -> Report DSL`
- `Report DSL -> DocumentArtifact`

## 2. 领域对象

- `TemplateInstance`
- `ReportDsl`
- `ReportInstance`
- `DocumentArtifact`
- `ExportJob`

`ReportInstance` 是资源包络；`content` 保存完整 `Report DSL`。

实现约束补充：

- `TemplateInstance` 和 `ReportDsl` 必须是递归 dataclass 聚合，不能只在顶层定义 dataclass、内部仍用 `dict/list[dict]`
- `TemplateInstance`、`ReportDsl` 及其子对象内部属性名保持 `snake_case`，对外公开字段名统一依赖 `field(metadata={"alias": "lowerCamelCase"})`
- `TemplateInstance.template` 必须组合完整 `ReportTemplate` dataclass 快照
- `TemplateInstance.structureType` 缺省为 `flow`；flow 实例使用 `catalogs`，paged 实例使用 `chapters -> slides -> sections`
- `ReportDsl.structureType` 缺省为 `flow`；flow DSL 使用 `catalogs + layout`，paged DSL 使用 `content`
- 本轮补齐 paged Report DSL 核心模型与 schema；`BuildReportDslService` 仍只编译 flow 结构，模板 paged 到 PPT DSL 的映射后续实现
- `ReportInstance.report` 必须组合完整 `ReportDsl` dataclass
- 入库、出库、导出时才允许做 JSON 序列化

## 3. 应用服务职责

当前代码中，报告运行时的应用层由 `ReportRuntimeService` 和 `ReportDocumentService` 组成。

### 3.1 `ReportRuntimeService`

承担以下功能职责：

- `persist_template_instance`
  - 新建或更新 `TemplateInstance`
  - 保证对话层始终围绕同一份模板实例推进
- `get_latest_template_instance`
  - 按会话读取最近一次模板实例快照
- `generate_report_from_template_instance`
  - 读取模板实例与模板快照
  - 构建 `Report DSL`
  - 校验 DSL
  - 创建 `ReportInstance`
  - 把模板实例状态推进到 `completed/report_ready`
- `get_report_view`
  - 组装 `/reports/{reportId}` 聚合视图
- `generate_documents`
  - 生成 Word/PPT/PDF/Markdown 产物
  - 创建 `ExportJob` 和 `DocumentArtifact`
- `resolve_download`
  - 解析 report-scoped 下载请求
- `serialize_report_answer`
  - 统一 `/chat` 与 `/reports` 的报告答案载荷

### 3.2 `ReportDocumentService`

承担以下功能职责：

- 把下载能力收束为 report-scoped 子服务
- 不暴露独立 document 资源集合

## 4. BuildReportDslService 规则

输入：

- 完整 `TemplateInstance`
- 模板快照
- 数据查询结果
- LLM 生成结果

输出：

- 满足 `design/report_system/schemas/report-dsl.schema.json` 的合法 `Report DSL`

必须完成：

1. 读取 `structureType` 对应的实例树；当前 DSL 编译实现读取 flow 的 `catalogs -> (subCatalogs)* -> sections`，paged 后续接入
2. 保留 `TemplateInstance.section.content` 的实例化视图
2. 根据 `runtimeContext.bindings` 生成查询表达式
3. 解析 `section.content.datasets`
4. 将 `presentation.blocks` 编译为 `components`
5. 填充新版 `basicInfo / layout / reportMeta / summary`
6. 校验 DSL

补充规则：

- flow 输出必须显式写入 `structureType = flow`，但 `catalogs` 分支保持 PPT 扩展前的旧契约：`Catalog.name`、`Section.summary`、`Cover.title` 等字段不因 paged 能力改变
- paged 输出使用 `content`，内容元素只能是 `ReportSlide[]` 或 `ReportSlideSection[]`，不得混放
- `reportMeta` 仍按全局唯一 `sectionId` 挂载
- `reportMeta[sectionId]` 对齐 BI Engine `GenerateMeta`：公开输出只包含 `status/question/additionalInfos/outline/parameters`
- `reportMeta[sectionId].additionalInfos` 使用复数字段名，条目必须包含 `type/value`，可包含 `name/appendix`；模型层兼容读取旧 `additionalInfo` 与 `content`
- `reportMeta[sectionId].parameters` 使用 DSL 专用参数结构 `id/label/required/widget/options`，不直接输出模板参数对象
- `basicInfo` 模型需支持 BI Engine 资产字段，包括 `schemaVersion/mode/subTitle/templateId/templateName/remark/createDate/modifyDate/creator/modifier/header/footer/category`
- `cover.layoutTemplate` 严格输出 `TITLE_TOP | TITLE_CENTER`，不再输出旧示例值 `default`
- `backCover` 作为 paged/PPT 封底配置挂在 `ReportDsl.back_cover`，公开字段为 `backCover`
- `layout.autoLayout`、表格列 `sortable/filterable/width`、`mergeColumns[].isMergeValue`、图表 `series/options.responsive` 与组件 `basicProperties/advanceProperties` 均作为 DSL 序列化边界透传字段
- 表格行合并模型统一为 `MergeRowInfo`，公开字段仍为 `startRowIndex/rowSpan/mergedText/column`
- 图表轴配置输出到 `ChartDataProperties.x_axis/y_axis`，公开字段为 `dataProperties.xAxis/yAxis`；旧顶层输入只做读取兼容
- 章节摘要继续输出到 `sections[].summary`，如需作为生成证据则写入 `reportMeta[sectionId].additionalInfos[]` 的 `Summary` 项
- 目录树位置由正式 `catalogs/subCatalogs/sections` 结构表达
- 生成进度主指标按 section 统计，目录级上下文通过 `generationProgress.currentCatalogPath` 辅助表达
- 参数与诉求要素的占位符渲染默认读取 `label` 通道；显式指定时才读取 `.value / .query`
- 模板实例化阶段必须解释 `dynamic`：
  - `foreach` 保持同一模板按参数值重复展开
  - `foreachCase` 按每个参数值匹配 case，并生成该 case 中定义的目录或章节内容
  - `custom` 不在实例化阶段展开，只在目录、章节或分页页面实例上保留 `dynamicContext.type/url/nodeType`
- 新实例必须输出 `dynamicContext`；`foreach/foreachCase` 记录 `type/parameterId/itemValue/caseId`，`custom` 记录 `type/url/nodeType`；`nodeType` 取值为 `catalog | section | slide`；旧 `foreachContext` 仅作为历史数据读取兼容
- Report DSL 冻结阶段遇到 custom 节点时，通过 runtime infrastructure 的外部内容 gateway 以 POST 调用 `url`；正式 v6 请求体包含 `parameters/templateNode/context`，旧 `nodeType/nodeId/prompt` 协议不再作为实现目标
- custom 目录响应按 `status/dsl/meta.dslType=Catalog` 解析，并用 flow `ReportCatalog` 片段替换当前目录，响应必须包含 `name`
- flow custom 章节响应按 `status/dsl/meta.dslType=Section` 解析，并用 flow `ReportSection` 片段替换当前章节，允许包含旧 `summary` 字段
- paged custom slide 响应按 `status/dsl/meta.dslType=Slide` 解析，并用 `ReportSlide` 片段替换当前 slide；paged slide 内 custom section 推荐返回 `dslType=Components` 并合并到当前 slide 组件集合，也允许返回 `Section` 后转换
- 当前实现已建立模板/schema/实例态 `nodeType=slide` 契约；HTTP gateway 请求体从旧协议迁移到 v6 以及 paged DSL 合并在后续运行时实现中完成
- `text` block 在实例化阶段必须把模板态 `properties.template` 渲染为实例态 `properties.content`，生成 DSL 时编译为 `TextComponent`
- 普通 `table` block 必须编译为 DSL `TableComponent`，透传 `datasetId`、`properties.columns`、`properties.mergeColumns` 与可计算的 `properties.mergeRows`
- `chart` block 必须编译为 DSL `ChartComponent`，透传 `datasetId`
- `properties.preferredType`、普通表格 `properties.columns/showTitle/defaultDisplayRows`、复合表 `tableLayout.columns/showTitle/defaultDisplayRows` 已进入模板定义和后端核心模型；运行时 DSL 编译透传不在本轮实现范围，后续实现阶段再补齐
- `composite_table` 是正式模板块类型，必须被编译为 DSL `compositeTable`
- `composite_table` 的 `query part` 必须把 `part.tableLayout.mergeColumns` 透传到对应子表 `dataProperties.mergeColumns`
- 表格 `dataProperties.data` 已存在时，运行时必须按 `mergeRows` 定义计算 DSL `dataProperties.mergeRows[]`，字段为 `startRowIndex/rowSpan/mergedText/column`
- `summary part` 当前固定编译为无表头静态两列表，运行时只填每行内容
- `TemplateInstance.section.content` 不是派生缓存；实例化阶段就必须落库，供二次编辑和重新生成复用
- `query part` 最小运行态固定为 `status/resolvedDatasetId/resolvedQuery/warnings`
- `summary part` 最小运行态固定为 `status/resolvedPartIds/prompt/warnings`

## 5. 文档导出规则

- 只接受 `Report DSL`
- `word/ppt` 由 Java 导出器生成
- `pdf` 从 `word` 或 `ppt` 派生
- 产物和任务状态分别落 `tbl_report_documents`、`tbl_export_jobs`

## 6. 对外接口映射

- `GET /reports/{reportId}`
- `POST /reports/{reportId}/document-generations`
- `GET /reports/{reportId}/documents/{documentId}/download`

`GET /reports/{reportId}` 返回的 `answer` 结构必须与 `/chat` 里 `REPORT.answer` 完全等价。

## 7. 与流式 `delta` 的边界

- `report_runtime` 只负责产出完整 `Report DSL`
- 目录与章节的流式 `delta` 属于 `conversation` 的对外投影，不写回 `ReportInstance`
- `ReportInstance` 与 `/reports/{reportId}` 始终只返回完整报告，不回放生成阶段的 patch 历史
