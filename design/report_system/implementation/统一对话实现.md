# 统一对话实现

## 1. 模块定位

`conversation` 负责统一对话协议下的报告生成交互，不承载报告正式内容定义。它的核心职责是维护：

- `Conversation`
- `Chat`
- `TemplateInstance`
- `ask / answer / steps / stream events`

## 2. TemplateInstance 的中心地位

每个报告生成会话必须围绕一份 `TemplateInstance` 推进。对话层需要做到：

1. 首轮匹配模板后创建模板实例
2. 将参数提取、结构化回填、默认值回填统一写入 `parameters[*].values` 与 `parameters[*].runtimeContext`
3. 基于模板实例当前状态决定下一轮 `ask`
4. 接收前端回传的完整模板实例树并整体覆盖到最新实例状态
5. 在 `confirm_params` 时把最终模板实例提交给报告运行时
6. 模板实例中的 `section.content` 必须与章节诉求一起返回，不能在对话层裁掉

补充规则：

- `extract_report_template` 返回的 `REPORT_TEMPLATE` 与 `/templates/import/preview` 共享同一套规范化结果与告警码
- 为支持历史报告的复杂二次编辑，对话层必须返回完整 `TemplateInstance`，而不是极简运行态骨架
- 对话层不得依赖“重新读取当前模板定义”来补齐历史实例缺失字段
- ask/reply 中涉及参数与诉求的对象，应逐步收敛到统一渐进式模型，而不是继续维持模板态/实例态/接口态多套异构结构
- 结构化 `reply` 必须带 `sourceChatId`，用于精确指向被消费的原始追问消息
- `reply.parameters` 只提交参数值映射，不回传完整参数定义；服务端基于当前 `TemplateInstance` 重新组装参数运行态
- 参数值三元组正式统一为 `{label, value, query}`；对话层不再接受或生成 `display`

## 3. 运行阶段

- `draft`
- `collecting_parameters`
- `ready_for_confirmation`
- `confirmed`
- `generating`
- `completed`
- `failed`

对话返回的 `status` 仍使用统一对话资源状态：

- `waiting_user`
- `running`
- `finished`
- `failed`

## 4. Ask/Answer 生成规则

- 参数未完备时返回 `ask`
- 参数完备且用户确认后返回 `answer.answerType=REPORT`
- 模板提取走 `instruction=extract_report_template`，返回 `REPORT_TEMPLATE`
- `ask.reportContext.templateInstance` 与 `answer.templateInstance` 都返回完整模板实例
- `ask.status` 用于表达这条追问消息是否已被后续回复消费：
  - 首次返回追问时为 `pending`
  - 用户成功提交对应 `reply` 后，原追问消息回显刷新为 `replied`

## 4.1 流式事件规则

- 流式 `/chat` 不再透传单个完整 `ChatResponse`，而是输出正式 `ChatStreamEvent`
- 事件分工固定为：
  - `status`：当前链路状态
  - `ask`：追问载荷
  - `answer`：最终完整结果
  - `done`：流结束
- 报告生成中的内容增量通过 `delta` 附着在 `answer` 事件上，首版动作只有：
  - `init_report`
  - `add_catalog`
  - `add_section`
- `delta` 只表达目录和章节 patch，不复用来传完整 `report/templateInstance`

## 5. 应用服务职责

当前代码中，对话模块的应用层主入口是 `ConversationService`。它内部集中承担以下 service 语义：

### 5.1 会话容器服务

- `list_sessions`
  - 读取会话列表和最近消息摘要
- `get_session`
  - 组装单个会话与完整聊天流水
- `delete_session`
  - 删除会话容器与聊天记录
- `fork_session`
  - 从指定聊天节点复制一条新会话

### 5.2 对话主流程服务

- `send_message`
  - 根据 `instruction` 分发报告生成或模板提取能力
- `_extract_report_template`
  - 走模板导入预览链路，返回 `REPORT_TEMPLATE`
- `_generate_report`
  - 负责 `generate_report` 主流程
  - 创建用户聊天消息
  - 读取或创建当前 `TemplateInstance`
  - 参数追问、参数确认和报告生成三段式推进

### 5.3 模板匹配与参数抽取服务

- `_match_template`
  - 组合词法匹配和 embedding 相似度，选出候选模板
- `_extract_parameter_values`
  - 根据模板参数定义，从用户问题中抽取 date / enum / dynamic / free_text 值
  - 抽取结果统一生成 `{label, value, query}` 三元组
  - `dynamic` 参数候选值匹配以 `label/value/query` 为唯一正式比较面

### 5.4 Ask/Answer 构建服务

- `_build_ask_response`
  - 基于模板实例当前填充状态，决定返回 `fill_params` 或 `confirm_params`
- `_append_assistant_message`
  - 把统一 `ChatResponse` 落入聊天流水
- `_reply_parameter_values_to_value_map`
  - 把 `reply.parameters: Record<parameterId, Scalar[]>` 重建为参数运行态值三元组
  - 严格依赖当前 `TemplateInstance` / 模板参数定义校验参数 id 与多值结构
- `mark_ask_replied`
  - 当用户成功提交带 `sourceChatId` 的 `reply` 后，只把 `sourceChatId` 指向的追问消息刷新为 `ask.status = replied`
  - 若 `sourceChatId` 缺失、跨会话、跨用户，或目标消息已不是 `pending`，则本轮 `reply` 直接视为非法请求
- `StreamChatEventService`
  - 负责把单次对话结果拆分成 `status / ask / answer / done` 事件
  - 在报告生成链路上额外附着 `delta`

对话应用层禁止承担：

- 直接生成 `Report DSL`
- 直接生成文档文件
- 提供独立 `TemplateInstance` 资源接口

目标态上，`ConversationService` 后续应继续向以下职责边界收敛：

- `SendChatMessageService`
- `TemplateMatchService`
- `ResolveParameterStateService`
- `ResolveDynamicParameterOptionsService`
- `BuildAskPayloadService`
- `ConfirmTemplateInstanceService`
- `StreamChatEventService`

## 6. 明确不做

- 不提供独立 `template_instance` 资源接口
- 不在对话层生成正式文档
- 不使用旧 `session/message` 命名
