# 统一对话实现

## 1. 模块定位

`conversation` 负责统一对话协议下的报告生成交互，不承载报告正式内容定义。它的核心职责是维护：

- `Conversation`
- `Chat`
- `TemplateInstance`
- `ask / answer / steps / stream events`

## 2. TemplateInstance 的中心地位

每个报告生成会话必须围绕一份 `TemplateInstance` 推进。对话层需要做到：

1. 首轮匹配模板后创建模板实例
2. 将参数提取、结构化回填、默认值回填统一写入 `parameters[*].values` 与 `parameters[*].runtime`
3. 基于模板实例当前状态决定下一轮 `ask`
4. 接收前端回传的完整模板实例树并更新 `deltaViews`
5. 在 `confirm_params` 时把最终模板实例提交给报告运行时

补充规则：

- 多层目录下，`deltaViews` 应尽量补充 `catalogPath/sectionPath`
- `extract_report_template` 返回的 `REPORT_TEMPLATE` 与 `/templates/import/preview` 共享同一套规范化结果与告警码
- 为支持历史报告的复杂二次编辑，对话层必须返回完整 `TemplateInstance`，而不是极简运行态骨架
- 对话层不得依赖“重新读取当前模板定义”来补齐历史实例缺失字段
- ask/reply 中涉及参数与诉求的对象，应逐步收敛到统一渐进式模型，而不是继续维持模板态/实例态/接口态多套异构结构

## 3. 运行阶段

- `draft`
- `collecting_parameters`
- `ready_for_confirmation`
- `confirmed`
- `generating`
- `completed`
- `failed`

对话返回的 `status` 仍使用统一对话资源状态：

- `waiting_user`
- `running`
- `finished`
- `failed`

## 4. Ask/Answer 生成规则

- 参数未完备时返回 `ask`
- 参数完备且用户确认后返回 `answer.answerType=REPORT`
- 模板提取走 `instruction=extract_report_template`，返回 `REPORT_TEMPLATE`
- `ask.reportContext.templateInstance` 与 `answer.templateInstance` 都返回完整模板实例

## 5. 应用服务职责

当前代码中，对话模块的应用层主入口是 `ConversationService`。它内部集中承担以下 service 语义：

### 5.1 会话容器服务

- `list_sessions`
  - 读取会话列表和最近消息摘要
- `get_session`
  - 组装单个会话与完整聊天流水
- `delete_session`
  - 删除会话容器与聊天记录
- `fork_session`
  - 从指定聊天节点复制一条新会话

### 5.2 对话主流程服务

- `send_message`
  - 根据 `instruction` 分发报告生成或模板提取能力
- `_extract_report_template`
  - 走模板导入预览链路，返回 `REPORT_TEMPLATE`
- `_generate_report`
  - 负责 `generate_report` 主流程
  - 创建用户聊天消息
  - 读取或创建当前 `TemplateInstance`
  - 参数追问、参数确认和报告生成三段式推进

### 5.3 模板匹配与参数抽取服务

- `_match_template`
  - 组合词法匹配和 embedding 相似度，选出候选模板
- `_extract_parameter_values`
  - 根据模板参数定义，从用户问题中抽取 date / enum / dynamic / free_text 值

### 5.4 Ask/Answer 构建服务

- `_build_ask_response`
  - 基于模板实例当前填充状态，决定返回 `fill_params` 或 `confirm_params`
- `_append_assistant_message`
  - 把统一 `ChatResponse` 落入聊天流水

对话应用层禁止承担：

- 直接生成 `Report DSL`
- 直接生成文档文件
- 提供独立 `TemplateInstance` 资源接口

目标态上，`ConversationService` 后续应继续向以下职责边界收敛：

- `SendChatMessageService`
- `TemplateMatchService`
- `ResolveParameterStateService`
- `ResolveDynamicParameterOptionsService`
- `BuildAskPayloadService`
- `ConfirmTemplateInstanceService`
- `StreamChatEventService`

## 6. 明确不做

- 不提供独立 `template_instance` 资源接口
- 不在对话层生成正式文档
- 不使用旧 `session/message` 命名
