# Report Template Schema 说明文档

> 对应文件：`design/report_system/schemas/report-template.schema.json`
> Schema 标题：`ReportTemplate`
> JSON Schema 版本：Draft 2020-12
> 本文用于解释报告模板 schema 的设计含义、字段结构、约束规则，并给出一份完整示例及逐段说明。

---

## 1. Schema 总览

这份 schema 定义的是**报告模板（ReportTemplate）**的静态结构。

从 schema 的描述可以看出：

> 报告模板正式定义。模板定义是静态系统资产，不带运行态状态字段。

也就是说，它描述的是一个“可复用的报告生成模板”，而不是一次报告生成任务的运行态数据。

它主要解决以下问题：

1. 报告模板的基本元信息是什么；
2. 报告需要哪些参数；
3. 报告目录如何组织；
4. 每个目录下有哪些章节；
5. 每个章节希望生成什么内容；
6. 每个章节需要查询哪些数据集；
7. 数据最终以什么形式展示；
8. 参数如何被标题、诉求、SQL、API 或展示块引用。

从整体结构看，一个模板大致可以理解为：

```text
ReportTemplate
├── 模板元信息
├── structureType           报告结构类型，缺省为 flow
├── parameters              全局参数
├── catalogs                瀑布流报告目录树，flow 使用
    ├── parameters          目录局部参数，可选
    ├── dynamic             目录动态展开，可选
    ├── subCatalogs         子目录，可选
    └── sections            章节，可选
        ├── parameters      章节局部参数，可选
        ├── dynamic         章节动态展开，可选
        ├── outline         章节生成诉求
        └── content         章节数据与展示定义
            ├── datasets
            └── presentation
└── chapters                分页报告章节树，paged 使用
    └── slides              分页页面
        ├── layout          页面布局提示，可选
        └── sections        页面内容块，复用 SectionDefinition
```

## 2. 顶层 ReportTemplate 结构

### 2.1 必填字段

顶层对象必须包含以下字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 模板唯一标识 |
| `category` | string | 模板分类 |
| `name` | string | 模板名称 |
| `description` | string | 模板描述 |
| `schemaVersion` | string | schema 或模板结构版本 |
| `parameters` | array | 模板级全局参数 |

`catalogs` 不再是无条件必填字段，而是受 `structureType` 控制：

- `structureType` 缺省或为 `flow` 时，必须提供 `catalogs`，且不能提供 `chapters`。
- `structureType = paged` 时，必须提供 `chapters`，且不能提供 `catalogs`。

生成后的 Report DSL 也使用同名 `structureType`：

- flow DSL 使用 `catalogs + layout`。
- paged/PPT DSL 使用 `content`，内容数组只能整体为 `Slide[]` 或整体为 `SlideSection[]`。
- 当前运行时仍只把模板实例编译为 flow DSL；paged 模板到 PPT DSL 的编译后续实现。

顶层对象设置了：

```json
"additionalProperties": false
```

这意味着顶层不允许出现 schema 未定义的额外字段。

例如，下面这个字段不合法：

```json
{
  "extraField": "not allowed"
}
```

### 2.2 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `structureType` | `"flow" | "paged"` | 报告结构类型，缺省为 `flow` |
| `createdAt` | string | 创建时间，格式为 `date-time` |
| `updatedAt` | string | 更新时间，格式为 `date-time` |

`createdAt` 和 `updatedAt` 使用 JSON Schema 的 `format: date-time`，建议使用 ISO 8601 格式，例如：

```json
"2026-04-24T10:00:00+08:00"
```

### 2.3 分页报告结构

当模板面向 PPT 等分页报告时，使用：

```json
{
  "structureType": "paged",
  "chapters": [
    {
      "id": "chapter_overview",
      "title": "整体概览",
      "slides": [
        {
          "id": "slide_kpi_overview",
          "title": "核心指标概览",
          "subtitle": "{$report_date}",
          "layout": {
            "layoutId": "title_content",
            "variant": "kpi_grid",
            "notes": "设计器可据此选择标题加指标卡片版式。"
          },
          "sections": []
        }
      ]
    }
  ]
}
```

`ChapterDefinition` 字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 分页章节唯一标识 |
| `title` | string | 分页章节标题，可选 |
| `description` | string | 分页章节说明，可选 |
| `implicit` | boolean | 是否为系统隐式生成的默认章节 |
| `parameters` | array | 分页章节局部参数，可选 |
| `dynamic` | object | 分页章节动态展开定义，可选 |
| `slides` | array | 页面列表，必填 |

`SlideDefinition` 字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 页面唯一标识 |
| `title` | string | 页面标题，可选 |
| `subtitle` | string | 页面副标题，可选 |
| `description` | string | 页面说明，可选 |
| `parameters` | array | 页面局部参数，可选 |
| `dynamic` | object | 页面动态展开定义，可选 |
| `layout` | object | 页面布局提示，可选 |
| `sections` | array | 页面内容块，复用 `SectionDefinition`，必填 |

默认隐式章节规范：

```json
{
  "id": "__default__",
  "title": "",
  "implicit": true,
  "slides": []
}
```

`SlideLayout` 当前只作为设计器和后续 PPT 渲染适配的提示字段，允许 `layoutId/variant/notes`，本轮不承诺运行时渲染语义。

### 2.4 id 命名规则

顶层 `id` 必须符合正则：

```regex
^[A-Za-z0-9_\-]+$
```

即只能包含：

- 英文字母；
- 数字；
- 下划线 `_`；
- 中划线 `-`。

合法示例：

```text
sales_report
sales-report-001
Report_2026
```

不合法示例：

```text
sales report
销售报告
sales.report
```

## 3. 参数定义 Parameter

`parameters` 是模板中非常核心的部分。参数可以出现在三个层级：

1. 顶层 `ReportTemplate.parameters`；
2. 目录 `CatalogDefinition.parameters`；
3. 章节 `SectionDefinition.parameters`。

这三类参数结构相同，都是 `Parameter`。

### 3.1 Parameter 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 参数唯一标识 |
| `label` | string | 参数展示名 |
| `inputType` | string | 参数输入类型 |
| `required` | boolean | 是否必填 |
| `multi` | boolean | 是否多选 |
| `interactionMode` | string | 参数交互方式 |

### 3.2 Parameter 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `description` | string | 参数说明 |
| `priority` | integer | 参数追问优先级，范围 `0-99`；数字越小优先级越高，缺省按 `99` |
| `placeholder` | string | 输入提示 |
| `defaultValue` | ParameterValue[] | 默认值 |
| `options` | ParameterValue[] | 枚举候选值 |
| `values` | ParameterValue[] | 当前参数值 |
| `runtimeContext` | ParameterRuntimeContext | 运行态上下文 |
| `source` | string | 动态候选值来源 URL |

### 3.3 inputType

`inputType` 只能是以下四种：

| 值 | 含义 | 典型用途 |
|---|---|---|
| `free_text` | 自由文本 | 用户输入任意文本 |
| `date` | 日期 | 单日或日期范围，具体由业务约定 |
| `enum` | 静态枚举 | 从固定候选项中选择 |
| `dynamic` | 动态枚举 | 从接口或外部来源加载候选值 |

### 3.4 interactionMode

`interactionMode` 只能是：

| 值 | 含义 |
|---|---|
| `form` | 表单输入 |
| `natural_language` | 自然语言补充 |

这表示模板参数既可以由前端表单收集，也可以由用户通过自然语言表达后由系统抽取。

### 3.5 priority

`priority` 控制对话生成报告时的参数追问顺序：

- 取值范围是 `0-99` 的整数；
- 数字越小优先级越高，`0` 是最高优先级，`99` 是最低优先级；
- 字段可选，缺省按 `99` 处理；
- 缺失必填参数按优先级从小到大追问，相同优先级的一批参数一起追问；
- `priority = 99` 的参数不单独追问，只在最终确认所有参数的环节一起展示和补齐；
- `priority` 不改变 `required` 语义，最终确认通过前所有必填参数仍必须有值。

### 3.6 enum 参数的约束

当：

```json
"inputType": "enum"
```

时，schema 要求必须提供：

```json
"options": [...]
```

示例：

```json
{
  "id": "region",
  "label": "区域",
  "inputType": "enum",
  "required": true,
  "multi": true,
  "interactionMode": "form",
  "options": [
    {
      "label": "华东",
      "value": "east",
      "query": "'east'"
    },
    {
      "label": "华南",
      "value": "south",
      "query": "'south'"
    }
  ]
}
```

### 3.7 dynamic 参数的约束

当：

```json
"inputType": "dynamic"
```

时，schema 要求必须提供：

```json
"source": "..."
```

示例：

```json
{
  "id": "store",
  "label": "门店",
  "inputType": "dynamic",
  "required": true,
  "multi": true,
  "interactionMode": "form",
  "source": "/api/stores"
}
```

这里的 `source` 被描述为“动态候选值来源地址。内容为 URL”。

## 4. 参数值 ParameterValue

`ParameterValue` 是参数取值的统一结构，要求包含：

| 字段 | 类型 | 说明 |
|---|---|---|
| `label` | Scalar | 给用户看的展示值 |
| `value` | Scalar | 系统内部使用的值 |
| `query` | Scalar | 查询语句或数据源中使用的值 |

`Scalar` 可以是：

- string；
- number；
- integer；
- boolean。

### 4.1 label、value、query 的区别

推荐这样理解：

| 字段 | 面向对象 | 示例 |
|---|---|---|
| `label` | 用户、LLM、报告标题、章节诉求 | `门店A` |
| `value` | 系统内部识别 | `store_001` |
| `query` | SQL/API/数据查询 | `'store_001'` |

例如：

```json
{
  "label": "门店A",
  "value": "store_001",
  "query": "'store_001'"
}
```

在报告标题中通常引用 `label`：

```text
{$store} 销售分析
```

在 SQL 中通常引用 `query`：

```sql
where store_id in ({$store})
```

## 5. 参数运行态上下文 ParameterRuntimeContext

`ParameterRuntimeContext` 用于记录参数在运行过程中的上下文信息。

虽然顶层 schema 描述说模板是静态系统资产，不带运行态状态字段，但当前 schema 仍然在 `Parameter` 中保留了 `runtimeContext` 字段。这意味着它可能用于：

1. 参数填充过程中的中间态；
2. 模板编辑、预览或调试；
3. 生成任务中复用同一套 Parameter 结构。

字段如下：

| 字段 | 类型 | 说明 |
|---|---|---|
| `valueSource` | string | 参数值来源 |
| `queryContext` | object | 查询上下文，可扩展 |
| `confirmed` | boolean | 是否已确认 |
| `confirmedAt` | string | 确认时间 |
| `optionSource` | string | 候选项来源 |
| `optionsFetchedAt` | string | 候选项拉取时间 |

`valueSource` 可选值：

| 值 | 含义 |
|---|---|
| `user_input` | 用户输入 |
| `default` | 默认值 |
| `dynamic_candidate` | 动态候选值 |
| `parameter_ref` | 引用其他参数 |
| `system_fill` | 系统自动填充 |

## 6. 目录 CatalogDefinition

`catalogs` 是报告目录树。每个目录节点由 `CatalogDefinition` 定义。

### 6.1 CatalogDefinition 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 目录 ID |
| `title` | string | 目录标题模板 |

### 6.2 CatalogDefinition 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `description` | string | 目录描述 |
| `parameters` | Parameter[] | 目录局部参数 |
| `dynamic` | DynamicDefinition | 动态生成目录或分支内容 |
| `subCatalogs` | CatalogDefinition[] | 子目录 |
| `sections` | SectionDefinition[] | 章节 |

### 6.3 sections 与 subCatalogs 的约束

`CatalogDefinition` 使用了：

```json
"anyOf": [
  { "required": ["sections"] },
  { "required": ["subCatalogs"] }
]
```

这意味着每个目录至少要有：

- `sections`；
- 或 `subCatalogs`；
- 或两者都有。

也就是说，一个目录不能只是空标题，它必须承载章节，或继续包含子目录。

### 6.4 title 中的参数引用

`title` 的描述是：

> 目录标题模板。可直接在句子中使用 `{$parameterId}` 引用当前层级及父层级可见参数的 label 值。

示例：

```json
"title": "{$region} 区域销售分析"
```

如果参数 `region` 当前值为：

```json
{
  "label": "华东",
  "value": "east",
  "query": "'east'"
}
```

则标题渲染后可以是：

```text
华东 区域销售分析
```

注意：schema 本身只规定 `title` 是字符串，并没有通过正则强制校验 `{$parameterId}` 是否存在。这类引用有效性通常需要业务校验器额外检查。

## 7. DynamicDefinition 动态展开定义

`dynamic` 用于根据参数取值动态生成目录或章节。它取代旧的 `foreach` 字段，是目录和章节动态展开的统一入口。

### 7.1 type 类型

| type | 说明 |
|---|---|
| `foreach` | 对同一目录或章节模板按参数值重复展开 |
| `foreachCase` | 按参数值循环展开，但不同值可命中不同 case 内容 |
| `custom` | 通过外部 URL 生成当前目录或章节的 Report DSL 片段 |

示例：

```json
"dynamic": {
  "type": "foreach",
  "parameterId": "store",
  "as": "currentStore"
}
```

含义是：对 `store` 参数中的多个值逐个循环，每次循环的当前值命名为 `currentStore`。

### 7.2 foreachCase

`foreachCase` 适用于“仍然按参数多值循环，但不同取值要生成不同章节或目录内容”的场景。

```json
"dynamic": {
  "type": "foreachCase",
  "parameterId": "scope",
  "as": "scope_item",
  "cases": [
    {
      "id": "hq_network_case",
      "values": ["hq-network"],
      "sections": []
    },
    {
      "id": "branch_network_case",
      "values": ["branch-network"],
      "subCatalogs": [],
      "sections": []
    }
  ],
  "defaultCase": {
    "sections": []
  }
}
```

### 7.3 custom

`custom` 适用于目录或章节内容由外部服务生成的场景。模板中必须声明 `url`：

```json
{
  "type": "custom",
  "url": "https://report-content.example.com/api/report/section"
}
```

- 目录级 custom 响应必须是新版 Report DSL `Catalog` 片段，使用 `title` 作为目录展示名，请求 `prompt` 使用目录实例 `renderedTitle`。
- 章节级 custom 响应必须是新版 Report DSL `Section` 片段，包含 `components`，不再返回旧 `summary` 字段；模板仍必须提供 `outline`，以便用户编辑大纲；请求 `prompt` 使用编辑后的 `outline.renderedRequirement`。
- 请求体包含 `nodeType`、`nodeId`、`parameters`、`prompt`；`parameters` 按当前节点可见参数的 `parameterId` 分组。
- `custom` 不再支持占位 `config` 字段。

规则：

1. case 匹配使用 `ParameterValue.value`，不使用 `label/query`；
2. `cases[].values` 可包含多个值，任一命中即使用该 case；
3. 多选参数按用户选择值逐个展开，多个值命中同一 case 时仍生成多次；
4. 未命中 case 时使用 `defaultCase`；没有 `defaultCase` 时跳过该参数值；
5. 目录级 case 可定义 `subCatalogs` 和/或 `sections`；
6. 章节级 case 通过 `sections` 定义章节变体，用于替换当前占位 section。

### 7.3 使用场景

例如用户选择了多个门店：

```json
"values": [
  { "label": "门店A", "value": "store_a", "query": "'store_a'" },
  { "label": "门店B", "value": "store_b", "query": "'store_b'" }
]
```

模板可以通过 `dynamic.type = foreach` 生成两个目录：

```text
门店A 销售分析
门店B 销售分析
```

## 8. 章节 SectionDefinition

`SectionDefinition` 定义报告中的具体章节。

### 8.1 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 章节 ID |
| `outline` | OutlineDefinition | 章节诉求定义 |
| `content` | SectionContentDefinition | 章节内容定义 |

### 8.2 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `description` | string | 章节描述 |
| `parameters` | Parameter[] | 章节局部参数 |
| `dynamic` | DynamicDefinition | 动态生成章节或章节变体 |

### 8.3 Section 的职责

一个章节通常包含两层信息：

1. `outline`：告诉系统“这一节要分析什么”；
2. `content`：告诉系统“为了完成分析，需要哪些数据，以及最终怎么展示”。

可以理解为：

```text
outline = 生成目标 / 分析诉求
content = 数据准备 + 呈现方式
```

## 9. 章节诉求 OutlineDefinition

`outline` 是智能报告生成中非常关键的结构，它负责描述章节的分析需求。

### 9.1 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `requirement` | string | 章节诉求模板 |
| `items` | RequirementItem[] | 诉求项定义 |

### 9.2 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `renderedRequirement` | string | 渲染后的诉求文本 |

### 9.3 requirement 与 items 的关系

`requirement` 是一段自然语言模板，可在文本中使用：

```text
{@itemId}
```

引用 `items` 中定义的诉求项。

例如：

```json
{
  "requirement": "分析 {@timeRange} 内 {@targetStore} 的 {@metric} 表现，并找出 {@topN} 个销售额最高的商品。",
  "items": [
    {
      "id": "timeRange",
      "label": "时间范围",
      "kind": "time_range",
      "required": true
    },
    {
      "id": "targetStore",
      "label": "目标门店",
      "kind": "parameter_ref",
      "required": true,
      "sourceParameterId": "store"
    },
    {
      "id": "metric",
      "label": "销售额",
      "kind": "metric",
      "required": true
    },
    {
      "id": "topN",
      "label": "Top 10",
      "kind": "threshold",
      "required": false
    }
  ]
}
```

`items` 的作用是把自然语言中的“槽位”结构化。
`requirement` 是给 LLM 或报告生成器看的完整诉求；
`items` 是给系统做参数收集、校验、渲染和追问用的结构化定义。

## 10. 诉求项 RequirementItem

`RequirementItem` 是 `outline.items` 中的元素。

### 10.1 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 诉求项 ID |
| `label` | string | 展示名或默认表达 |
| `kind` | string | 诉求项类型 |
| `required` | boolean | 是否必填 |

### 10.2 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `multi` | boolean | 是否多值 |
| `description` | string | 说明 |
| `sourceParameterId` | string | 来源参数 ID |
| `widget` | string | 前端组件 |
| `defaultValue` | ParameterValue[] | 默认值 |
| `values` | ParameterValue[] | 当前值 |
| `valueSource` | string | 值来源 |

### 10.3 kind 可选值

| kind | 含义 | 示例 |
|---|---|---|
| `search_target` | 查询目标 | 商品、客户、站点 |
| `search_condition` | 查询条件 | 满足某条件的订单 |
| `metric` | 指标 | 销售额、利润率、客流量 |
| `time_range` | 时间范围 | 近 30 天、本月 |
| `filter` | 过滤条件 | 区域、品类、渠道 |
| `threshold` | 阈值 | Top 10、超过 80% |
| `sort` | 排序方式 | 按销售额降序 |
| `free_text` | 自由文本 | 其他分析要求 |
| `parameter_ref` | 引用参数 | 引用全局参数 `store` |

### 10.4 widget 可选值

| widget | 说明 |
|---|---|
| `input` | 单行输入框 |
| `textarea` | 多行输入框 |
| `select` | 单选 |
| `multi_select` | 多选 |
| `date` | 日期 |
| `date_range` | 日期范围 |

### 10.5 valueSource 可选值

| valueSource | 含义 |
|---|---|
| `user_input` | 用户输入 |
| `default` | 默认值 |
| `parameter_ref` | 参数引用 |
| `system_fill` | 系统填充 |

### 10.6 requirement 引用 label，SQL 引用 query

在模板设计中，需要区分两个层面的引用：

1. `outline.requirement` 主要面向自然语言表达，通常使用参数或诉求项的 `label`；
2. `content.datasets.source` 中的 SQL/API 模板主要面向数据查询，通常使用参数值的 `query`。

例如参数：

```json
{
  "label": "门店A",
  "value": "store_a",
  "query": "'store_a'"
}
```

在自然语言诉求中应渲染为：

```text
分析 门店A 的销售表现
```

在 SQL 中应渲染为：

```sql
where store_id = 'store_a'
```

因此，`label` 和 `query` 不建议混用。

## 12. 数据集 DatasetDefinition

`DatasetDefinition` 描述章节需要的数据来源。

### 12.1 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 数据集 ID |
| `sourceType` | string | 数据来源类型 |
| `source` | string | 数据源定义字符串 |

### 12.2 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `name` | string | 数据集名称 |
| `dependsOn` | string[] | 依赖的数据集 ID |
| `description` | string | 数据集说明 |

### 12.3 sourceType 可选值

| sourceType | 含义 |
|---|---|
| `sql` | SQL 查询模板 |
| `api` | API URL |
| `llm` | LLM 生成或加工 |
| `compose` | 组合多个数据集 |

### 12.4 source 字段的含义

`source` 被设计成统一字符串：

- 当 `sourceType = sql` 时，`source` 是 SQL 模板；
- 当 `sourceType = api` 时，`source` 是 API URL；
- 当 `sourceType = llm` 或 `compose` 时，也沿用字符串口径。

示例：

```json
{
  "id": "ds_sales_top_products",
  "sourceType": "sql",
  "source": "select product_name, sum(sales_amount) as sales_amount from sales where store_id in ({$store}) and biz_date between {$startDate} and {$endDate} group by product_name order by sales_amount desc limit {$topN}"
}
```

其中：

- `{$store}` 可替换为参数值的 `query`；
- `{$startDate}`、`{$endDate}` 可替换为日期参数对应的查询值；
- `{$topN}` 可替换为阈值。

注意：schema 只规定 `source` 是字符串，不校验 SQL 语法，也不校验占位符是否存在。实际落地时应由模板校验器补充校验。

## 14. 展示块 PresentationBlock

`PresentationBlock` 是展示内容的最小配置单元之一。

### 14.1 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | string | 展示块 ID |
| `type` | string | 展示块类型 |

### 14.2 可选字段

| 字段 | 类型 | 说明 |
|---|---|---|
| `title` | string | 展示块标题 |
| `datasetId` | string | 绑定的数据集 ID |
| `properties` | PresentationProperty | 展示属性，text 支持 `template/content`；chart 支持 `preferredType`；普通 table 支持 `columns/showTitle/defaultDisplayRows/mergeColumns/mergeRows` |
| `description` | string | 展示块描述 |
| `parts` | CompositeTablePart[] | 复合表格组成部分 |

### 14.3 type 可选值

| type | 含义 |
|---|---|
| `text` | 文本 |
| `table` | 表格 |
| `chart` | 图表 |
| `composite_table` | 复合表格 |

`paragraph`、`bullet`、`kpi`、`markdown` 不再作为模板 presentation block 类型使用。

### 14.4 text 的特殊约束

当：

```json
"type": "text"
```

时，必须提供：

```json
"properties": {
  "template": "..."
}
```

`properties.template` 支持两类引用：

| 语法 | 含义 |
|---|---|
| `{$parameterId}` | 引用当前章节可见参数，面向文本展示时默认使用参数 `label` |
| `{#datasetId.field}` | 引用同一 section 内某个 dataset 执行结果第一行的字段值 |

一个 `properties.template` 可以同时引用多个参数和多个 dataset 字段。例如：

```json
{
  "id": "block_summary_text",
  "type": "text",
  "title": "经营结论",
  "properties": {
    "template": "{$region} 区域销售额为 {#ds_sales_summary.sales_amount}，订单数为 {#ds_sales_summary.order_count}。"
  }
}
```

其中：

- `ds_sales_summary` 必须是同一 section 的 `content.datasets[].id`；
- `sales_amount`、`order_count` 使用 dataset 执行结果中的字段 key，不使用展示名；
- 当前版本约束 dataset 结果按单行数据理解；如果实际返回多行，默认取第一行对应字段值；
- schema 只校验 `properties.template` 是字符串，不校验 `{#datasetId.field}` 是否真实存在。
- 实例态 text block 还必须在 `properties.content` 中保存渲染后的展示文本。

### 14.5 chart/table 的 properties

`properties` 是展示属性容器，不同 `type` 生效字段不同。

当 `type = chart` 时，支持：

| 字段 | 类型 | 说明 |
|---|---|---|
| `preferredType` | string | 图表首选展示类型，可选 `line`、`bar`、`pie`、`scatter`、`radar`、`gauge`、`candlestick` |

当 `type = table` 时，支持：

| 字段 | 类型 | 说明 |
|---|---|---|
| `columns` | TableColumn[] | 普通表格列定义，`key` 对应 dataset 字段 key，`title` 是展示名 |
| `showTitle` | boolean | 是否显示表格标题；只影响展示，不改变 `title` 字段本身 |
| `defaultDisplayRows` | integer | 默认展示数据条数；只作为展示提示，不截断 dataset 结果 |
| `mergeColumns` | MergeColumnInfo[] | 合并列定义，`columns` 内填写源数据列 key |
| `mergeRows` | MergeRowDefinition[] | 行合并定义，`column` 引用表格列 key |

示例：

```json
{
  "id": "block_summary_table",
  "type": "table",
  "title": "关键指标明细",
  "datasetId": "dataset_daily_summary",
  "properties": {
    "showTitle": true,
    "defaultDisplayRows": 10,
    "columns": [
      {"key": "region_name", "title": "范围"},
      {"key": "availability_rate", "title": "可用率"}
    ],
    "mergeRows": [
      {"column": "region_name"}
    ]
  }
}
```

### 14.6 composite_table 的特殊约束

当：

```json
"type": "composite_table"
```

时，必须提供：

```json
"parts": [...]
```

并且 `parts` 至少有一个元素。

## 17. 行合并 MergeRowDefinition

字段如下：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `column` | string | 是 | 目标列 key，对应 `TableComponent.dataProperties.columns[].key` |
| `mode` | string | 否 | 当前仅支持 `default`；缺省为 `default` |

`default` 模式要求数据已经按目标列排好序。报告生成时，如果 `TableComponent.dataProperties.data` 已存在，则按行顺序扫描 `column` 对应字段，把连续多行相同值合并为一个单元格，并在 Report DSL 输出 `dataProperties.mergeRows[]`：

```json
{
  "startRowIndex": 0,
  "rowSpan": 3,
  "mergedText": "总部网络",
  "column": "scope_name"
}
```

`startRowIndex` 从 0 起算；Report DSL 使用 `column`，不使用 `columnKey`。

## 20. 示例说明

### 20.1 顶层模板信息

示例中的顶层对象：

```json
{
  "id": "store_sales_analysis_report",
  "category": "sales",
  "name": "门店销售经营分析报告"
}
```

表示这是一个销售类报告模板。

`id` 是模板的系统唯一标识，`name` 是用户可见名称，`description` 说明模板用途。

### 20.2 全局参数

示例定义了四个全局参数：

| 参数 ID | 含义 | 输入类型 | 是否多选 | 说明 |
|---|---|---|---|---|
| `region` | 区域 | `enum` | 是 | 静态枚举 |
| `store` | 门店 | `dynamic` | 是 | 从接口动态加载 |
| `dateRange` | 分析时间范围 | `date` | 否 | 可通过自然语言补充 |
| `topN` | 榜单数量 | `free_text` | 否 | 默认 Top 10 |

其中 `region` 是枚举参数，因此必须提供 `options`。

`store` 是动态参数，因此必须提供 `source`。

`dateRange` 使用了：

```json
"interactionMode": "natural_language"
```

表示用户可以说“分析近 30 天”或“看一下本月”，系统再抽取成结构化参数。

### 20.3 目录结构

示例有两个一级目录：

```text
华东 区域销售概览
门店明细分析
```

第一个目录直接包含 `sections`。
第二个目录包含 `subCatalogs`，并通过 `dynamic.type = foreach` 对 `store` 参数循环生成门店子目录。

### 20.4 dynamic.foreach 的作用

在：

```json
"dynamic": {
  "type": "foreach",
  "parameterId": "store",
  "as": "currentStore"
}
```

中，表示对用户选择的多个门店逐个展开。

如果用户选择了：

```text
门店A、门店B、门店C
```

则可以生成：

```text
门店A 门店销售分析
门店B 门店销售分析
门店C 门店销售分析
```

schema 本身只定义了循环结构，不定义展开后的运行态结构。展开逻辑应由报告生成引擎实现。

### 20.5 outline 与 content 的分工

以 `section_top_products` 为例：

`outline.requirement` 是：

```text
分析 {@timeRange} 内 {@targetStore} 的商品销售表现，筛选 {@categoryFilter}，找出 {@topN} 个销售额最高的商品，并说明主要贡献来源。
```

这段内容描述“要分析什么”。

而 `content.datasets` 中的 SQL 描述“怎么查数据”：

```sql
select product_id, product_name, category_name, sum(sales_amount) as sales_amount
from dwd_sales_order_item
where store_id in ({$store})
  and {$dateRange}
  and category_id in ({$productCategory})
group by product_id, product_name, category_name
order by sales_amount desc
limit {$topN}
```

`content.presentation` 描述“怎么展示”：

- 表格展示 Top 商品；
- 图表展示销售额对比；
- 要点列表生成关键发现。

### 20.6 RequirementItem 中的 parameter_ref

示例中大量使用：

```json
"kind": "parameter_ref",
"sourceParameterId": "store"
```

表示这个诉求项不是单独收集的，而是引用已有参数。

例如：

```json
{
  "id": "targetStore",
  "label": "当前门店",
  "kind": "parameter_ref",
  "required": true,
  "sourceParameterId": "store",
  "valueSource": "parameter_ref"
}
```

它在自然语言诉求中用于表达：

```text
当前门店
```

实际渲染时应结合 `store` 参数当前循环值或当前参数值。

### 20.7 title、requirement、source 的引用差异

示例中有三类常见引用：

#### 1. 目录标题引用

```json
"title": "{$region} 区域销售概览"
```

这里面向用户展示，推荐渲染参数的 `label`。

#### 2. 章节诉求引用

```text
分析 {@timeRange} 内 {@targetStore} 的商品销售表现
```

这里引用的是 `outline.items` 的 ID，最终也偏向自然语言表达，推荐渲染为 label 或可读文本。

#### 3. SQL source 引用

```sql
where store_id in ({$store}) and {$dateRange}
```

这里面向数据查询，推荐渲染参数的 `query`。

#### 4. properties.template 引用

```text
{$region} 区域销售额为 {#ds_sales_summary.sales_amount}
```

这里面向最终文本展示：

- `{$region}` 仍是参数引用，推荐渲染参数的 `label`；
- `{#ds_sales_summary.sales_amount}` 是 dataset 字段引用，读取同一 section 中 `ds_sales_summary` 执行结果第一行的 `sales_amount` 字段值。

因此可以形成一条清晰规则：

```text
标题 / 诉求：使用 label
SQL / API 查询：使用 query
properties.template：参数使用 label，dataset 字段使用 {#datasetId.field}
系统内部流转：使用 value
```

### 20.8 properties.template 的 dataset 字段引用

`text` block 适合承载短结论、摘要、指标解读。除了参数引用外，它可以引用一个或多个 dataset 的字段：

```json
{
  "id": "block_summary_text",
  "type": "text",
  "title": "经营结论",
  "properties": {
    "template": "{$region} 区域销售额为 {#ds_sales_summary.sales_amount}，Top 商品为 {#ds_top_products.product_name}。"
  }
}
```

规则如下：

1. `{#datasetId.field}` 中的 `datasetId` 必须指向同一 section 的 `content.datasets[].id`；
2. `field` 使用 dataset 执行结果中的字段 key；
3. 一个 `template` 可以包含多个 `{#datasetId.field}`，也可以混合使用 `{$parameterId}`；
4. 当前版本按单行 dataset 结果理解；如果 dataset 返回多行，默认取第一行的对应字段值；
5. schema 不负责校验 dataset 或字段是否存在，建议由业务校验器补充。

### 20.9 复合表格说明

`section_store_composite_table` 展示了一个复合表格：

```text
门店经营复合分析表
├── 品类销售 query part
├── 环比对比 query part
└── 经营摘要 summary part
```

其中：

- `part_category_sales` 来自 `ds_store_category_sales`；
- `part_period_compare` 来自 `ds_store_period_compare`；
- `part_summary` 汇总前两个 part。

`summarySpec.partIds` 指明要汇总哪些 part：

```json
"partIds": [
  "part_category_sales",
  "part_period_compare"
]
```

`summarySpec.rows` 定义汇总结果的行：

```json
[
  {
    "id": "advantage",
    "title": "优势"
  },
  {
    "id": "risk",
    "title": "风险"
  },
  {
    "id": "suggestion",
    "title": "建议"
  }
]
```

这适合生成“数据 + 分析结论”混合型表格。

## 21. 使用建议与补充校验规则

当前 schema 已经定义了结构类型、必填字段、枚举值和部分条件约束。
但在真实系统中，建议增加一层业务校验，补充 schema 无法完全表达的规则。

### 21.1 引用完整性校验

建议校验以下引用是否存在：

| 引用位置 | 应校验内容 |
|---|---|
| `{$parameterId}` | 参数是否在当前层级或父层级可见 |
| `{@itemId}` | `outline.items` 中是否存在对应 item |
| `{#datasetId.field}` | dataset 是否存在于同一 section，字段 key 是否存在于执行结果 |
| `sourceParameterId` | 是否存在对应参数 |
| `datasetId` | 是否存在对应 dataset |
| `dependsOn` | 是否存在对应 dataset |
| `summarySpec.partIds` | 是否存在对应 composite table part |

### 21.2 参数作用域校验

建议明确参数作用域优先级：

```text
章节参数 > 目录参数 > 父目录参数 > 全局参数
```

当同名参数在不同层级出现时，需要规定覆盖策略。

### 21.3 dynamic 语义校验

建议校验：

1. `dynamic.parameterId` 是否存在；
2. `dynamic.type = foreach/foreachCase` 时，被循环参数是否 `multi = true`；
3. 循环变量 `as` 是否可以在标题、outline、SQL 中被引用；
4. 循环中使用 `{$store}` 时，是引用完整参数列表，还是当前循环值；
5. `foreachCase.cases[].values` 是否能匹配参数的 `ParameterValue.value`；
6. 章节级 `foreachCase` 的 case 是否提供合法的章节变体。

### 21.4 SQL 占位符校验

建议校验：

1. SQL 中所有 `{$xxx}` 是否存在对应参数；
2. 参数是否允许用于 SQL；
3. 参数值是否有 `query`；
4. 多选参数在 SQL 中的拼接规则；
5. 空值参数如何处理；
6. 可选过滤条件缺失时是否删除整段条件。

### 21.5 outline item 完整性校验

建议校验：

1. `requirement` 中的所有 `{@itemId}` 是否都在 `items` 中定义；
2. `items` 中定义但未被 requirement 引用的项是否允许；
3. `kind = parameter_ref` 时是否必须提供 `sourceParameterId`；
4. `valueSource = parameter_ref` 时是否必须提供 `sourceParameterId`；
5. `required = true` 的 item 是否最终有值。

### 21.6 presentation 与 dataset 绑定校验

建议校验：

1. `PresentationBlock.datasetId` 是否存在；
2. `type = table/chart` 是否必须有 `datasetId`；
3. `type = text` 是否必须有 `properties.template`；
4. `properties.template` 中 `{#datasetId.field}` 引用的 dataset 是否存在，字段 key 是否可由 dataset 结果提供；
5. 引用 dataset 多行结果时是否接受“默认取第一行”的 v1 规则；
6. `type = composite_table` 时是否不应再直接使用 `datasetId`；
7. `composite_table.parts` 中 query part 的 `datasetId` 是否存在。

### 21.7 dynamic 参数 source 校验

对动态参数，建议校验：

1. `source` 是否是合法 URL 或内部 API 路径；
2. `source` 中的 `{$parameterId}` 是否可解析；
3. 动态接口返回值是否能映射为 `ParameterValue[]`；
4. 动态候选是否需要缓存；
5. 参数依赖变更后是否需要重新拉取候选。

## 22. 推荐的模板生成流程

结合该 schema，一个报告生成流程可以设计为：

```text
1. 选择报告模板
2. 读取顶层 parameters
3. 根据 interactionMode 收集参数
   ├── form：前端表单
   └── natural_language：自然语言抽取
4. 解析目录 catalogs
5. 处理 dynamic，展开目录或章节
6. 渲染目录 title
7. 渲染 outline.requirement
8. 根据 content.datasets 执行数据查询
9. 根据 content.presentation 生成展示块
10. 汇总为最终报告
```

## 23. 总结

这份 schema 的核心设计可以概括为：

```text
ReportTemplate = 参数定义 + 目录结构 + 章节诉求 + 数据集 + 展示方式
```

其中：

- `parameters` 定义报告生成所需的输入；
- `catalogs` 定义报告目录树；
- `sections` 定义具体章节；
- `outline` 定义章节生成诉求；
- `items` 将自然语言诉求结构化；
- `datasets` 定义数据来源；
- `presentation` 定义展示形态；
- `composite_table` 支持复杂表格和汇总分析。

推荐在 schema 校验之外，额外实现一层业务校验器，用于校验参数引用、数据集引用、SQL 占位符、dynamic 作用域和展示块绑定关系。这样可以让模板既满足 JSON Schema 的结构合法性，也满足报告生成引擎的实际执行要求。
